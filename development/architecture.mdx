draw.io Desktop is a standard Electron application composed of two distinct processes — a **main process** and a **renderer process** — communicating through a tightly controlled IPC bridge.

## Two-process model

<CardGroup cols={2}>
  <Card title="Main process" icon="server">
    `src/main/electron.js` (2,700+ lines). Runs in Node.js. Owns the OS-level concerns: window management, file I/O, native menus, auto-update, and all IPC handlers.
  </Card>
  <Card title="Renderer process" icon="browser">
    The draw.io web application (`drawio/src/main/webapp/`). Runs in a sandboxed Chromium context with no direct Node.js access.
  </Card>
</CardGroup>

The renderer is loaded as a local `file://` URL pointing into the submodule:

```js
// src/main/electron.js
const codeDir = path.join(__dirname, '/../../drawio/src/main/webapp');

let ourl = url.format({
    pathname: `${codeDir}/index.html`,
    protocol: 'file:',
    query: queryObj,
    slashes: true
})

mainWindow.loadURL(ourl)
```

## The draw.io submodule

The `drawio/` directory is a **git submodule** pointing at [github.com/jgraph/drawio](https://github.com/jgraph/drawio). It contains the complete draw.io web editor — all HTML, JavaScript, and assets.

```text
drawio-desktop/
├── src/main/
│   ├── electron.js           # Main process
│   ├── electron-preload.js   # IPC bridge (preload script)
│   └── disableUpdate.js      # Generated by sync script
└── drawio/                   # Git submodule
    └── src/main/webapp/      # Web app loaded in every BrowserWindow
        └── index.html        # Entry point for the renderer
```

The submodule is the **source of truth for the version number**. `drawio/VERSION` is read by `sync.cjs` and written into `package.json` before each release build.

## Main process responsibilities

`electron.js` handles everything that requires Node.js or OS access:

- **Window management** — creates and tracks `BrowserWindow` instances in `windowsRegistry`; restores last window size/position from `electron-store`
- **File I/O** — all filesystem reads and writes on behalf of the renderer (save, open, watch, delete)
- **Native menus** — application menu on macOS; no menu bar on Windows/Linux
- **Auto-update** — checks GitHub releases on startup via `electron-updater`; shows a download/install dialog
- **IPC handling** — the single `ipcMain.on('rendererReq', ...)` listener dispatches all renderer requests
- **Security enforcement** — Content Security Policy injection and `validateSender()` origin checks on every IPC message

## Renderer process

The renderer is the draw.io web app running inside a `BrowserWindow`. It has no direct access to Node.js APIs — all OS interaction goes through the `window.electron` object exposed by the preload script.

### URL parameters (`queryObj`)

The main process passes configuration to the renderer via query string parameters on the `file://` URL. These are assembled in `queryObj` before `loadURL` is called:

```js
// src/main/electron.js
var queryObj = {
    'dev': __DEV__ ? 1 : 0,
    'gapi': 0,         // Google Drive disabled
    'db': 0,           // Dropbox disabled
    'od': 0,           // OneDrive disabled
    'gh': 0,           // GitHub disabled
    'gl': 0,           // GitLab disabled
    'mode': 'device',  // Local file mode
    'export': 'https://convert.diagrams.net/node/export',
    'disableUpdate': disableUpdate ? 1 : 0,
    'enableSpellCheck': enableSpellCheck ? 1 : 0,
    // ...
};
```

An optional `urlParams.json` file in the working directory can override individual parameters at startup.

## Preload script and the IPC bridge

`src/main/electron-preload.js` runs in a privileged context — it has access to both `ipcRenderer` (Node.js-side) and the renderer's `window` object. It uses Electron's `contextBridge` to expose a safe, narrow API surface to the web app:

```js
// src/main/electron-preload.js
contextBridge.exposeInMainWorld('electron', {
    request: (msg, callback, error) => { ... },
    registerMsgListener: (action, callback) => { ... },
    sendMessage: (action, args) => { ... },
    listenOnce: (action, callback) => { ... }
});
```

The renderer calls `window.electron.request(...)` — it never touches `ipcRenderer` directly.

See [IPC communication](/development/ipc) for the full protocol.

## Security model

draw.io Desktop is explicitly designed to be **isolated from the internet** (except for the update check). Several layers enforce this:

<CardGroup cols={2}>
  <Card title="contextIsolation: true" icon="shield">
    The renderer's JavaScript context is completely separated from the preload script's context. The renderer cannot access Node.js APIs even if it tries.
  </Card>
  <Card title="No nodeIntegration" icon="ban">
    `nodeIntegration` is never enabled. The renderer has no `require` and no direct filesystem access.
  </Card>
  <Card title="validateSender()" icon="lock">
    Every IPC message arriving at the main process is checked: the sender's frame URL must start with the local `codeUrl`. Requests from any other origin are silently dropped.
  </Card>
  <Card title="Content Security Policy" icon="file-shield">
    The main process injects a strict CSP on all responses, forbidding remote script execution and external data transmission.
  </Card>
</CardGroup>

### `validateSender()` in detail

```js
// src/main/electron.js
const codeUrl = url.pathToFileURL(codeDir).href
    .replace(/\/.\:\//, str => str.toUpperCase()); // Windows drive letter fix

function validateSender(frame) {
    return frame.url
        .replace(/\/.\:\//, str => str.toUpperCase())
        .startsWith(codeUrl);
}
```

This is called at the top of every `ipcMain.on` handler. If the check fails, the handler returns `null` immediately without processing the request.

### Content Security Policy

The CSP is enforced via the session's `onHeadersReceived` hook:

```js
// src/main/electron.js
session.defaultSession.webRequest.onHeadersReceived((details, callback) => {
    callback({
        responseHeaders: {
            ...details.responseHeaders,
            'Content-Security-Policy': [
                "default-src 'self'; script-src 'self'; connect-src 'self'" +
                (isGoogleFontsEnabled ? ' https://fonts.googleapis.com https://fonts.gstatic.com' : '') +
                "; img-src * data:; media-src *; font-src * data:; frame-src 'self'; style-src 'self' 'unsafe-inline'" +
                (isGoogleFontsEnabled ? ' https://fonts.googleapis.com' : '') +
                "; base-uri 'none';child-src 'self';object-src 'none';"
            ]
        }
    });
});
```

### File access restriction

A `webRequest.onBeforeRequest` filter blocks any `file://` request that does not originate from within `codeUrl` or the optional plugins directory. This prevents the renderer from reading arbitrary files from the filesystem.

## Persistent settings

`electron-store` provides a simple key-value store backed by a JSON file in the OS app data folder. It persists:

- Last window size and position (`lastWinSize`)
- Spell check preference (`enableSpellCheck`)
- Auto-save/backup preference (`enableStoreBkp`)
- Google Fonts toggle (`isGoogleFontsEnabled`)
- Update opt-out flag (`dontCheckUpdates`) — set when the user dismisses an update prompt
